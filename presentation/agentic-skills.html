¿Qué son las Skills?

Las skills son capacidades especializadas que se agregan a Claude Code para extender su funcionalidad. Son instrucciones y contexto pre-empaquetado que le dicen al agente exactamente cómo realizar tareas específicas.

🎨 edslides Crea presentaciones HTML
🔍 review Revisa pull requests
⚙️ update-config Configura hooks y settings
🛡️ security-review Audita vulnerabilidades
🔧 gopls Inteligencia para Go
📦 init Documenta el codebase
Se invocan con /nombre-de-la-skill directamente en la conversación con el agente.

¿Qué son las Agent Skills?

Las Agent Skills son skills avanzadas que permiten a Claude actuar como un agente autónomo. Ejecutan tareas complejas de múltiples pasos, coordinan herramientas, toman decisiones y completan workflows end-to-end sin intervención manual constante.

🧠

Autonomía

Toma decisiones y adapta su plan según el contexto

🔗

Multi-paso

Encadena herramientas y acciones en secuencia lógica

⚡

Paralelismo

Coordina subtareas independientes de forma concurrente

🎯

End-to-End

Completa workflows completos sin intervención continua

¿Qué soluciona?

sin skills

Cada vez que inicias una conversación, tienes que explicarle de nuevo al agente cómo quieres que realice la tarea: el formato, el tono, los pasos, las reglas…

"Hazlo así…" "Recuerda que…" "Como te dije…"
→
con skills

El agente ya sabe cómo realizar la tarea de manera efectiva. El conocimiento está empaquetado en la skill: no hay que repetirlo en cada llamado.

✓ Consistencia en cada ejecución
✓ Cero fricción al invocar
✓ Conocimiento reutilizable y compartible

¿Cómo se usa?

📋

La descripción decide todo

El agente lee la descripción de cada skill y determina automáticamente si debe usarla para resolver la tarea. No hace falta invocarla manualmente.

⚡

Se carga bajo demanda

El contenido de la skill solo se inyecta al contexto cuando el agente decide que la necesita. Si no es relevante para la tarea, no se carga.

CLAUDE.md y Agents.md

CLAUDE.md / Agents.md

Reglas del proyecto, siempre presentes

  • ✓ Se cargan en todas las conversaciones automáticamente
  • ✓ Permanecen en el contexto durante toda la sesión
  • ✓ Definen reglas, convenciones y restricciones del proyecto
  • ~ Consumen tokens en cada conversación, sin excepción
vs
Skills

Capacidades especializadas, bajo demanda

  • ✓ Se cargan solo cuando la tarea las requiere
  • ✓ El agente infiere cuándo aplicarlas según el contexto
  • ✓ Empaquetan conocimiento experto para tareas específicas
  • ~ No están disponibles si el agente no las detecta como relevantes
Los archivos CLAUDE.md son el contexto base; las skills son el conocimiento especializado que se activa según la necesidad.

SubAgentes

Los SubAgentes son instancias de Claude que ejecutan trabajos de forma aislada e independiente. Son los encargados de ejecutar las skills cuando el agente principal los invoca.

01

Contexto aislado

Cada subagente tiene su propio contexto. No comparte estado con el agente principal, evitando interferencias.

02

Ejecución paralela

Múltiples subagentes pueden correr simultáneamente, acelerando workflows complejos con tareas independientes.

03

Ejecutores de skills

Cuando el agente principal invoca una skill, delega la ejecución a un subagente especializado para completarla.

Hooks

Los hooks son comandos shell que se ejecutan automáticamente en respuesta a eventos del sistema. A diferencia de las skills, no las infiere el agente: se disparan por el harness.

PostToolUse → Luego de llamar una herramienta (Bash, Edit, Write…)
Stop → Al terminar de procesar un prompt completo
PreToolUse → Antes de ejecutar cualquier herramienta
Hooks Se disparan por eventos — deterministas, siempre ocurren si la condición se cumple
|
Skills Se infieren en la conversación — el agente decide cuándo son relevantes

Comandos personalizados

Comandos personalizados

Invocación explícita por el usuario

  • ✓ Se ejecutan con /nombre-del-comando de forma explícita
  • ✓ El usuario tiene control total sobre cuándo se ejecutan
  • ✕ Los agentes no pueden invocarlos — solo el usuario humano
  • ~ Útiles para flujos manuales y comandos de uso frecuente
vs
Skills

Invocación inferencial por el agente

  • ✓ El agente las detecta e invoca sin intervención del usuario
  • ✓ Se activan por inferencia según el contexto de la tarea
  • ✓ Los agentes pueden llamarlas dentro de workflows automáticos
  • ~ El usuario también puede invocarlas con /nombre si lo desea
La diferencia clave: los comandos personalizados son herramientas del usuario; las skills son herramientas del agente.

Directory structure

# Ubicación según el agente: .claude/skills/ · .codex/skills/ · .opencode/skills/ · etc. skill-name/ ├── SKILL.md # Required: metadata + instructions ├── scripts/ # Optional: executable code │ └── process.py ├── references/ # Optional: documentation │ └── documentation.md ├── assets/ # Optional: templates, resources │ └── image.jpg └── ... # Any additional files or directories
scripts/

Código ejecutable

Scripts que el agente puede correr. Auto-contenidos, con errores claros y manejo de edge cases. Soporta Python, Bash, JavaScript…

references/

Documentación adicional

Archivos que el agente lee bajo demanda: REFERENCE.md, FORMS.md, o documentos de dominio. Mantenlos pequeños y enfocados.

assets/

Recursos estáticos

Templates de documentos o configuración, imágenes (diagramas, ejemplos) y archivos de datos (schemas, lookup tables).

Frontmatter · obligatorias

name Required Máx. 64 caracteres. Solo minúsculas a-z, números y guiones. No empieza ni termina en -, ni dos guiones seguidos. Debe coincidir con el nombre del directorio.
name: pdf-processing
description Required Máx. 1024 caracteres. Describe qué hace la skill y cuándo usarla. Incluye keywords que ayuden al agente a identificar tareas relevantes.
description: Extracts text and tables from PDF files, fills PDF forms, and merges multiple PDFs. Use when working with PDF documents or when the user mentions PDFs, forms, or document extraction.
Una buena description es lo único que decide si tu skill se activa o no.

Frontmatter · opcionales

license Optional Nombre de la licencia o referencia a un archivo de licencia incluido. Recomendado mantenerlo corto.
license: Proprietary. LICENSE.txt has complete terms
compatibility Optional Máx. 500 caracteres. Solo si tu skill tiene requisitos de entorno: producto destino, paquetes del sistema, acceso a red…
compatibility: Requires git, docker, jq, and access to the internet
metadata Optional Mapa de claves y valores string para propiedades extra no definidas por la spec. Usa nombres de claves únicos para evitar colisiones.
metadata: author: example-org version: "1.0"
allowed-tools Optional Lista separada por espacios de herramientas pre-aprobadas que la skill puede usar. Experimental — el soporte varía entre agentes.
allowed-tools: Bash(git:*) Bash(jq:*) Read

AGENTS.md — el README para agentes de IA

  • Qué es: formato abierto y estandarizado que da contexto e instrucciones a agentes de IA en proyectos de código (un “README para agentes”).
  • Compatibilidad: soportado por más de 25 herramientas — OpenAI Codex, GitHub Copilot, Google Jules, Gemini CLI, Cursor, VS Code, Zed, Aider, JetBrains Junie, Devin, Windsurf, Warp.
  • Ubicación: archivo en la raíz del repositorio; en monorepos se pueden anidar varios AGENTS.md y el agente lee el más cercano al archivo editado.
  • Precedencia: el AGENTS.md más cercano gana sobre los superiores; los prompts explícitos del usuario anulan al archivo.
  • Formato: Markdown estándar, sin campos obligatorios ni esquema rígido.

Secciones recomendadas

  • Descripción general del proyecto
  • Comandos de construcción (build)
  • Comandos y procedimientos de testing
  • Directrices de estilo de código
  • Convenciones de mensajes de commit
  • Convenciones para pull requests
  • Consideraciones de seguridad

Ejemplo: AGENTS.md para un proyecto en Go

# AGENTS.md ## Proyecto API REST en Go 1.23 con arquitectura hexagonal. Base de datos PostgreSQL, migraciones con goose. ## Testing - `go test -race -cover ./...` con detector de race y cobertura - Tests de integración requieren Postgres: `make test-integration` ## Estilo de código - Formatear con `gofmt` y `goimports` antes de commitear - Linter: `golangci-lint run` debe pasar sin errores - Nombres de paquetes en minúsculas, sin guiones bajos - Errores: envolver con `fmt.Errorf("contexto: %w", err)` - Todos los ID de las struct deben ser uuid de `google/uuid` versión 4 - Siempre utiliza `gopls` para verificar que el código está limpio y actualizado ## Convenciones de commits - Formato: `tipo(scope): descripción` (ej. `feat(auth): agrega JWT`) - Tipos: feat, fix, refactor, test, docs, chore ## Seguridad - Nunca loguear tokens, contraseñas ni datos PII - Usar `crypto/rand` para generar secrets, nunca `math/rand` - Validar entradas en la capa de handlers

SkillsBench: ¿qué tanto sirven las Skills?

  • Qué es: benchmark que mide el impacto de las Agent Skills en LLMs sin modificar el modelo.
  • Curadas vs. auto-generadas: las humanas suben +16.2 pp; las auto-generadas no aportan o restan.
  • Por dominio: Salud +51.9 pp y Manufactura +41.9 pp; Software (+4.5 pp) y Matemáticas (+6.0 pp), bajo.
  • Menos es más: skills de 2–3 módulos ganan a los manuales extensos.
  • Escala vs. skills: un modelo pequeño con skills puede superar a uno grande sin ellas.
  • Top configs: Gemini 3 Flash + Gemini CLI lidera con 48.7%; Claude Opus 4.5 logra la mayor mejora (+23.3 pp).
